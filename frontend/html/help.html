// tips

smart query The LLM generates an optimised Elasticsearch DSL query before searching. Best for specific, structured questions. Disable for broad keyword searches or if the model is slow.
concise Limits the response to 3–5 sentences. Useful for quick triage when you don't need full detail.
results slider Controls how many log records are retrieved and sent to the LLM. 20–50 is a good default. Use all results only for aggregation queries — it can be very slow.
indices Select only the indices relevant to your question. For authentication, use windows.eventlogs.rdpauth or windows.eventlogs.evtx. For network, use windows.network.netstat or windows.network.netstatenriched_netstat.
per-host queries Data is collected from multiple hosts (DC01, DC02, WKS01–WKS05, FILESVR). Narrow by asking "on DC01" or "across all workstations" — the LLM will filter by host.name.
ECS fields All data is normalised to Elastic Common Schema. Use source.ip, destination.ip, user.name, process.name, host.name in your questions — the LLM understands these directly.
[ eql ] After any query runs, click [ eql ] on the ES query block to convert it to Event Query Language — useful for importing into Elastic Security detection rules.
show full prompt Enable in the sidebar to see exactly what is sent to the LLM — both the query generation phase (field names + your question) and the analysis phase (ES results + your question). Useful for debugging unexpected answers.
thinking models deepseek-r1 and similar models show their reasoning before answering. The reasoning block auto-collapses when the answer starts. Best for complex multi-step investigations.

// authentication & logon events

Show me all failed login attempts and which accounts were targeted
Queries event.outcome:failure in RDPAuth and Evtx (EventID 4625). Groups by user.name to show most targeted accounts.
List usernames with both successful and failed logins, and how many of each
Aggregates EventID 4624 (success) and 4625 (failure) per user. Accounts with many failures followed by a success may indicate a brute force that worked.
Which source IPs had RDP logon attempts?
Uses source.ip from RDPAuth and Evtx. Identifies external IPs attempting remote desktop access.
Show me all logon type 3 (network) events and where they came from
Logon type 3 covers SMB, RDP, and network shares. Unusual source IPs or off-hours timing warrants investigation (MITRE T1021).
Are there logins from outside the 192.168.0.0/16 range?
Identifies external authentication attempts against internal systems. Any non-RFC1918 source IP for successful logins is high priority.
Which service accounts logged in interactively?
Service accounts (typically ending in $, or named svc_*) should not have interactive logon type 2 or 10 sessions. Indicates stolen credentials or misconfiguration.

// network connections

Show me all connections to external IP addresses
Filters destination.ip for non-RFC1918 addresses. Outbound connections to public IPs may indicate C2 (T1071) or data exfiltration (T1041).
Which processes have established connections and what are their destination IPs?
Joins process.name with destination.ip and destination.port from NetstatEnriched. Unusual processes with network connections are high priority.
Are there connections on unusual ports like 4444, 8080, or 1337?
Common C2 and reverse shell ports. Uses destination.port across Netstat indices.
Show me all ESTABLISHED connections from DC01 and group by destination IP
Filters by host.name:dc01 and Status:ESTABLISHED. Helps identify what a domain controller is actively communicating with.
Which processes are listening on the network and are they signed?
Uses NetstatEnriched to correlate listening services with code_signature.trusted. Unsigned listening processes are suspicious.
Find connections where the process hash is known malicious or unsigned
Checks Authenticode_Trusted and Hash_SHA256 fields from NetstatEnriched. Untrusted processes with external connections are critical findings.

// process execution & malware

Show me PowerShell commands that look suspicious
Searches ps_suspicious:true and ps_indicators fields across PowerShell script block and module logs. Flags encoded commands, download cradles, and known tool names.
Are there processes spawned from unusual parents like Word or Excel?
Uses process.parent.name from EventID 4688 records. Office apps spawning cmd.exe or PowerShell indicates a malicious macro (T1566).
Find processes running from temp or AppData directories
Malware frequently executes from user-writable paths. Checks process.executable for %TEMP%, AppData, Downloads.
Show me all processes created by the Administrator account
Uses user.name:Administrator with EventID 4688 (process create). Unexpected admin-run processes during off-hours are suspicious.
What processes ran on WKS01 that don't appear on other workstations?
Compares process.name across hosts. Unique processes on a single host can indicate targeted compromise.
Are there signs of credential dumping or LSASS access?
Looks for access to lsass.exe, ntds.dit, SAM database, and Mimikatz indicators in PowerShell logs and process creation events (T1003).

// lateral movement

Are there signs of lateral movement across the domain?
Broad sweep across RDPAuth, Evtx, and Netstat — covers pass-the-hash, Kerberos ticket reuse, SMB connections between workstations, and authentication chains (T1021, T1550).
Which accounts authenticated to multiple different hosts?
Aggregates user.name against host.name across all authentication indices. Accounts touching many systems in a short window may indicate automated lateral movement.
Show me RDP connections between workstations (not to/from domain controllers)
Peer-to-peer RDP between WKS hosts is unusual — legitimate admin RDP typically goes to DCs or servers, not between workstations (T1021.001).
Are any workstations connecting to the file server FILESVR on port 445?
SMB connections to file servers are normal, but unusual source hosts or times may indicate ransomware staging or data exfiltration via SMB (T1021.002).

// persistence & privilege escalation

Show me all autorun entries that are unsigned or from unusual locations
Queries the Autoruns index — checks Signer and file.path for persistence mechanisms in startup keys, run keys, and scheduled tasks (T1547, T1053).
Were any new user accounts created or modified recently?
Searches EventID 4720 (create), 4738 (modify), 4726 (delete) in the Evtx index. New privileged accounts are a common persistence technique (T1136).
Are there any scheduled tasks created by non-system accounts?
Uses EventID 4698 records — filters for user.name that isn't SYSTEM or a known admin. Attacker-created scheduled tasks are a reliable persistence mechanism (T1053.005).
Show me new services installed outside of business hours
EventID 4697 (service install) filtered by timestamp. Malicious services are often installed during off-hours to reduce detection window (T1543.003).
Which accounts were added to privileged groups?
EventID 4728/4732/4756 — group membership changes to Domain Admins, Administrators, or Remote Desktop Users are high-value events (T1098).

// file system & MFT

Show me executable files created in temp or AppData directories
Queries the NTFS MFT index for .exe, .dll, .ps1 files in user-writable paths. Malware staging almost always involves writing to these locations.
Find files with Alternate Data Streams (ADS)
ADS are used to hide payloads inside legitimate files (file.attributes.ads:true in MFT). A common defence evasion technique (T1564.004).
What scripts (ps1, bat, vbs) were created on the domain controller?
Filters MFT by host.name and file extension. Scripts created on DCs outside change windows are high-priority findings.
Show me files created within the last 24 hours of the collection time
Uses file.created timestamp from MFT with a date range filter. Useful for establishing what was dropped during an attack window.

// browser history

What external URLs did users visit from the domain controller?
DCs browsing the internet is unusual. Uses url.original and host.name from Chrome/Firefox history. Any external browsing from DC01/DC02 warrants investigation.
Show me downloads or file transfer URLs visited by any user
Searches url.original for file hosting services, pastebin, or direct download links. May indicate malware delivery (T1105) or data staging (T1567).
Which users visited IP addresses directly (not domain names)?
Direct IP browsing (e.g. http://1.2.3.4) bypasses DNS and is commonly used for C2 check-ins. Searches url.original for IP-format URLs.

// investigation & triage

Summarise all activity associated with IP address 192.168.111.101
Full cross-index sweep using source.ip and destination.ip. Shows authentication attempts, network connections, and any other events linked to that IP.
What did the user Administrator do across all hosts?
Queries user.name:Administrator across all indices. Reconstructs a full activity timeline — logons, processes, network connections, and file access.
Give me a timeline of everything that happened on DC01 on 2024-07-26
Uses host.name:dc01 with a date range on @timestamp. Pulls events from all relevant indices to reconstruct a full day's activity.
What are the most critical findings across all collected data?
High-level triage sweep — asks the LLM to identify the most suspicious indicators across all indices and rank them by severity. Good starting point for an investigation.
Map the suspicious activity to MITRE ATT&CK techniques
Takes events from previous queries and classifies them against the MITRE ATT&CK framework with tactic/technique IDs. Useful for writing IR reports.
Compare activity between DC01 and DC02 — are there differences that suggest one is compromised?
Baselining: unusual processes, services, or network connections on one DC that don't appear on the other may indicate targeted compromise.

// velociraptor collection context

data sources This data was collected by Velociraptor from: DC01, DC02 (domain controllers), FILESVR (file server), WKS01–WKS05 (workstations). All timestamps are UTC. Collection date: 2024-08-04.
available indices windows.eventlogs.evtx — all Windows event logs (Security, System, Application) · windows.eventlogs.rdpauth — RDP/logon events only · windows.network.netstat — point-in-time connections · windows.network.netstatenriched_netstat — connections with process + hash info · windows.ntfs.mft — file system (executables and suspicious paths only) · windows.sysinternals.autoruns — persistence locations · windows.eventlogs.powershellscriptblock — PowerShell activity
point-in-time Netstat data is a snapshot, not a stream — it shows connections active at collection time, not all connections that ever existed. For full network history, use authentication logs instead.
new log sources Drop new .json files into ./logs/ and they will be automatically ingested. Velociraptor collections, Sysmon JSONL, Zeek logs, and Suricata EVE JSON are all supported. ECS normalisation is applied automatically for known Velociraptor artefact types.
